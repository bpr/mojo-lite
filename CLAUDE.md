# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`mojo-lite` is a Rust (edition 2024; one dependency, `petgraph`) implementation of a growing subset of the [Mojo](https://www.modular.com/mojo) language — a lexer, parser, static type checker, and an **ownership-aware compiler backend**: the checked AST is lowered to a control-flow graph (HIR) and then flattened to A-Normal-Form MIR, over which ownership/move/drop analysis runs, and a **register VM** executes the verified MIR. (A tree-walking evaluator was the original executor; it has been retired — the register VM is the sole backend, behind a `trait Backend` seam kept for a future native backend.)

> **Note on this doc:** the per-feature bullets below describe language *semantics*, which the VM implements. Where an older bullet says "the evaluator" it now means "the VM / at runtime"; a few internals it named (`eval_call`, `Evaluator.output`, `update_place`) lived in the deleted tree-walker — the equivalents now live in `src/backend/vm.rs` and `src/runtime/mod.rs`. The compiler pipeline is described in the Architecture section.

**Core invariant: mojo-lite is a _strict subset of current Mojo_.** Every construct it accepts must be valid Mojo; the language may *restrict* Mojo (e.g. it requires type annotations Mojo would let you infer) but must never invent syntax Mojo lacks. The north star is real Mojo. Verify syntax against current Mojo docs before adding it — Mojo changes fast (e.g. `fn` was removed; `def` is the only function keyword; `let` was removed).

Currently implemented surface:

- **Variable declarations**: `var x: Int = 42` (annotated) or `var x = 42` (**inferred** — the annotation is optional; the type comes from the initializer, materializing a literal to its default kind so `var x = 5` is `Int`, `var f = 3.5` is `Float64`). `Stmt::VarDecl.ty` is `Option<Type>`. An inferred `var` of a value that can't live in a named binding is rejected: a closure (`ClosureEscape`) or the non-first-class `range` (see `inferred_binding_ty` in `checker.rs`, shared with the var-less `x = e` path). Scalar types: `Int` (i64), `UInt` (u64), `Float64` (f64), `Bool`, `String`, `None`. Every `var` is **mutable**.
- **Numbers**: `Int`/`UInt`/`Float64`. Two *concrete* numeric types never mix in one operator (`i + f` with `i: Int`, `f: Float64` is a type error — convert first). But **numeric literals coerce**: an int literal is flexible (→ `Int`/`UInt`/`Float64`), a float literal (`3.5`, `1e5`) → `Float64`, materializing to whatever the context wants. So `var u: UInt = 0`, `var f: Float64 = 3`, `u + 1`, and `1 / 2` (= `0.5`) all just work; a literal combined only with literals stays a literal (`var u: UInt = 1 + 2`). Explicit conversions `Int(x)` / `UInt(x)` / `Float64(x)` (one numeric-or-`Bool` arg; `Float64`→int truncates toward zero, `Bool` is 0/1) convert *between concrete* numeric types; like `range` they resolve in call position when not shadowed.
- **Assignment**: `x = expr` re-assigns an already-declared variable (distinct from the `var` *declaration*); the value must keep the variable's declared type. Assignment walks out to the nearest enclosing scope that defines the name and updates it there (so a loop/branch body can mutate an enclosing variable — that is what makes loops able to accumulate). Assigning a closure value is rejected (`ClosureEscape`), mirroring the no-escaping-closure rule for `return`. `x = expr` on an **undeclared** name is a Mojo `var`-less introduction (implicit declaration): the checker binds the implicit var with the value's materialized type, and the VM **runs** it as an ordinary binding (it lowers identically to `var x = expr`). *(The retired tree-walker deferred this as an unsupported "parse now, run later" gap; the compiler backend simply executes it.)*
- **Functions**: `def name(a: Int, b: Int) -> Int:` with an indented body; `pass` for an empty body; omitting `-> T` means the function returns `None`. Call by name: `f(1, 2)`. **Entry point**: after running a program's top-level statements, the VM calls a zero-argument `main()` if one is defined (Mojo-style), so a file whose work lives in `main()` actually runs. **Default parameter values** (`def my_pow(base: Int, exp: Int = 2)`) are **implemented** for non-generic free `def`s: defaults must be trailing (`required_count` — a required param after a defaulted one is rejected), a default value must fit its parameter's type, `Ty::Func` carries the `required` count so a call may supply `required..=params.len()` arguments (the evaluator fills omitted trailing params by evaluating each default in the function's captured scope). *Deferred:* defaults on generic functions and on methods (still flagged). **Keyword arguments** (`f(x=1, y=2)` → `KwArg`) are **implemented** for free-function calls: `Ty::Func` carries the parameter `names`, and the shared `match_call_slots` binds positional then keyword arguments to parameter slots (in both the checker and the evaluator, so they agree), filling defaults — reporting an unknown keyword, an argument bound twice, or a missing required argument as `TypeError::BadCall` (too-many-positional stays `ArityMismatch`, matching Python/Mojo). *Deferred:* keyword args to methods, generic functions, and built-ins (all flagged). A trailing **`*args` homogeneous variadic** (`def sum(*values: Int)`) is **implemented** for non-generic free `def`s: `Ty::Func` carries the variadic's *element* type, extra positional arguments beyond the regular params (each coerced to that element type) are collected into a `List[element]` bound in the body (so `for v in values`, `len(values)`, `values[i]` all work). `*args` must be the single, trailing parameter (a parameter after it would be keyword-only — deferred). The **full Mojo parameter grammar is parsed** (`FnParam`): `**kwargs` variadics (`ParamKind`), the `/` positional-only and `*` keyword-only markers (`positional_only`/`keyword_only` indices), and argument conventions (`read`/`mut`/`owned`/`out`/`ref` → `ArgConvention`; the **`ref`** convention may carry an **origin specifier** `ref[origin] x`, whose contents — an expression/`origin_of(…)`, a named origin, or `_` — are parsed and **discarded**). These remaining *signature* forms are **parse-only** — the checker flags any def using them as `TypeError::Unsupported` (a plain, defaults-only, keyword-called, or `*args` `def` is unaffected). A convention word is only a convention when a name follows it, so `read`/`mut`/`ref`/… stay usable as parameter names. A **reference type** `ref[origin] T` (e.g. a `-> ref[o] T` return) parses into `Type::Ref` (origin discarded) and is flagged by `ty_from_anno`; `origin_of(…)` itself is just an ordinary call. Named origin *parameters* (`def f[o: Origin[…]](…)`) and the `//` infer-only marker are **not** parsed (deeper, deferred).
- **Structs** (`@fieldwise_init struct Name:` with `var name: Type` fields and `def` methods): value types with typed fields and methods. `self` is the implicit, untyped first method parameter — **read-only** by default, or **writable** when the receiver is declared **`mut self`** (`def m(mut self):`), which permits `self.x = e` (and calling sibling `mut self` methods), with the mutation **written back** to the receiver so it persists (`c.increment()`). **Decorators use a general `@dotted.name[(args)]` grammar** (`Decorator { path, args, kwargs }`, parsed on `def`/`struct`/method, stacked) but **only `@fieldwise_init` is acted on** — it sets `Stmt::Struct.fieldwise_init` (derived in the parser from the decorator list), generating the constructor so `Name(v1, v2, ...)` builds an instance; a struct without it can't be constructed. Every other decorator (`@value`, `@staticmethod`, `@always_inline`, …) is recorded and ignored. **Dunder methods** (`__eq__`, `__add__`, `__init__`, …) parse as ordinary methods by name. The **receiver** is `[read|mut|owned|out|ref] self` (`Method.self_convention`; `ref self` may carry a discarded origin) or absent for a `@staticmethod` (`Method.has_self = false`); only `mut self` is modeled — the checker flags `out`/`owned`/`ref self` and `@staticmethod` (no self) as `Unsupported` (parsed, semantics deferred). Member read `obj.field`, method call `obj.method(args)`, and chaining (`a.b.c`, `p.m().n`) are parsed left-recursively in `primary`. **Member write** `p.x = e` and nested places (`p.a.b = e`, `p.items[i] = e`) — see the member-write note below. Struct *types* live in a registry (a separate namespace from value bindings); a non-builtin type annotation (`var p: Point`) is `Type::Named(name, args)` and the checker validates it names a defined struct (and its type-argument count). Deferred (parse-only): hand-written `__init__` (`out self`), operator overloading semantics, ordinary-param conventions (incl. `ref`), and `ref self` / reference types + origins (`ref[origin] T`, `origin_of(…)`).
- **Parameterization / generics (Phase 1a — type parameters)**: a `struct` or `def` may declare compile-time **type parameters** in `[...]`: `struct Pair[T: Copyable & Movable]:` / `def first[T: Copyable & Movable](p: Pair[T]) -> T:`. Only *type* parameters are supported (no value params `[n: Int]`, no `alias`/comptime). **Every parameter must carry a trait bound** — Mojo has no unconstrained parameters (bare `[T]` is a parse error); the least restrictive is `[T: AnyType]`. Bounds name a **built-in** trait (`AnyType`/`Copyable`/`Movable`/… — see `BUILTIN_TRAITS` in `checker.rs`) or a **user `trait`** (Phase 1b, below). A `T` bounded only by traits with no methods we model is **opaque**: you can store, pass, and return it, but no operator is defined on it (`x + x` on a `T` is a `BadOperator`) — only its bound traits' methods are callable. Inside a struct, refer to its own parameters as **`Self.T`** (Mojo spelling; bare `T` there is `UnknownType`); a field is `var left: Self.T`. Type arguments appear in annotations (`Pair[Int]`) but there is **no explicit `f[Int](...)` call syntax** — call sites **infer** each type parameter by unifying declared param/field types against the argument types (`unify`/`substitute` in `checker.rs`). Generics **type-erase at runtime**: a generic `def`/`struct` runs as ordinary dynamic code and the evaluator ignores `[...]` entirely — the checker does all the work. Soundness note: a numeric literal argument is **defaulted to its concrete type when solving a parameter** (so `Pair(1, 2)` is `Pair[Int]`), which deliberately forbids cross-argument literal widening (`Pair(1.0, 2)` is a *conflict*, not `Pair[Float64]`) — this keeps the solved type matching the value the type-erased evaluator actually stores. Deferred: value parameters + comptime + wider/SIMD numerics (2), method-level type params, and explicit call-site type arguments.
- **Value parameters + comptime (Phase 2)**: a parameter list may also declare **value parameters** — `struct FixedBuffer[size: Int]:` / `def repeat[count: Int](...)`. The `[name: X]` form is classified by the checker: `X` a trait ⇒ type parameter, `X` a concrete type ⇒ **value** parameter (restricted to `Int`). A value parameter is an `Int` **compile-time value** readable in the body — `Self.size` in a struct method, bare `n` in a function — and it is **reified onto the instance at runtime** (a `FixedBuffer[8]` value stores `size=8`; not pure type-erasure, unlike type params). It is part of the type's identity: `FixedBuffer[8]` and `FixedBuffer[9]` are distinct types. Value parameters may **not** appear in field/parameter/return **type** annotations (no dependent types) — only as a struct's own type-identity args and as body values. **Supplying parameters**: explicit `Name[args](callargs)` gives all parameters positionally (`FixedBuffer[8](0)`, `Box[Int](5)`); a value argument is a **comptime `Int` expression** — literals, `comptime` constants, and `+ - * // % **` / unary `-`, evaluated at check time (`Pair[2+3]` ⇒ `Pair[5]`). Omitting `[...]` still **infers** type parameters from arguments (Phase 1a), but a value parameter cannot be inferred (must be explicit). **`comptime NAME = expr`** declares a compile-time `Int` constant (Mojo removed `alias`; `comptime` replaces it — see [[mojo-comptime-replaced-alias]]), usable both as a value argument and as an ordinary runtime `Int`. **`comptime if cond:`** (compile-time conditional, with `elif`/`else`) and **`comptime for var in iter:`** (compile-time unrolled loop) are the modern Mojo spellings — the older `@parameter if`/`@parameter for` are deprecated, so (strict-subset) mojo-lite does not accept them; both are **parsed and grammar-documented** (`Stmt::ComptimeIf`/`ComptimeFor`, `parse_comptime` branching on the token after `comptime`; the `if`/`for` bodies reuse `parse_if_rest`/`parse_for_rest`) **but the checker flags them `Unsupported`** (comptime branch selection / unrolling deferred; parse-only per [[syntax-first-vm-compiler-roadmap]]). Deferred: value parameters of non-`Int` type, `comptime if`/`comptime for` *semantics*, dependent types, and SIMD/wider numerics.
- **Imports (parsed, not resolved)**: `import a.b.c [as alias]` and `from [.]*module import name [as alias], … | *`. Dotted module paths, `as` aliases, comma-separated names, wildcard `*`, and **relative** imports (leading dots set the level — `from .mod import X`, `from . import X`, `from ..pkg import X`; because the lexer makes `...` a single ellipsis, a 3-dot level comes through as one `...`). This is a deliberate **parse-now-run-later** feature (per [[parse-broadly-defer-semantics]]): mojo-lite has no module system, so the checker and evaluator treat an import as a **no-op** (imported names are *not* made available — using one is an `UndefinedVariable`). The eventual design (passing module environments, a `Module(env)` value) is future work. Keeps the parser a complete Mojo syntax front-end.
- **`List[T]` (collections)**: a built-in generic, growable, homogeneous sequence, and — like Mojo — a **value type** (assigning/passing **copies** it; no aliasing). Construct with `List[T]()` / `List[T](a, b, …)` (explicit element type) or `List(a, b, …)` / the literal `[a, b, …]` (element type **inferred** — numeric literals unify, so `[1, 2.0]` is `List[Float64]`; empty `[]`/`List()` can't infer, use `List[T]()`). `len(xs)`, index read `xs[i]` (`Int` index, `0..len`, runtime-checked; no negative indices yet), and `for x in xs:` iterate. **Mutating methods** (`append(e)`, `insert(i, e)`, `remove(e)`, `pop([i])`, `clear()`, `reverse()`, `extend(other)`) mutate the list *in place* at its **place** — a variable *or* a field/index chain rooted at one (`bag.items.append(e)`) — never a temporary/call result (a checker `MutationRequiresVariable`); see the member-write note. Value semantics hold: a copy is independent. **Read-only queries** `count(e)` / `index(e)` work on any list (variable or temporary; `list_query`), as does membership `e in xs` / `e not in xs` → `Bool`. `remove`/`count`/`index`/`in` compare elements, so they require an **equatable** element type (`is_list_equatable` — the scalar `==` set). Known cosmetic caveat: appending a numeric *literal* to an *empty* typed list (`List[Float64]().append(1)`) stores the literal's natural kind (there's no element to coerce against); reads coerce it.
- **Augmented assignment**: `target OP= e` (`+= -= *= /= //= %= **=`) means `target = target OP e` over the seven arithmetic operators. The target is a `NAME` or place (same rules as `=`); the operator/result must type-check like the expansion (so `i /= 2` on an `Int` is an error — `/` yields `Float64`). The place (and any index in it) is evaluated **once** — a real `Stmt::AugAssign` node doing a read-modify-write through the place machinery, not parse-time desugaring, so `xs[f()] += 1` calls `f` once. `and`/`or`/comparison/`in` have no augmented form (they aren't arithmetic).
- **`Tuple[T1, …, Tn]` (tuples)**: a built-in **fixed-size, heterogeneous** value type. Type `Tuple[Int, Float64, String]`; literal `(a, b, …)` (`()` empty, `(a,)` a 1-tuple; a bare `(e)` is grouping). Element read `t[i]` where **`i` is a compile-time constant** (a literal or `comptime`) — required because elements have different types, so the index must be statically known to give `t[i]`'s type (`eval_ct` in `infer_index`; a runtime index or an out-of-range one is a checker `TypeMismatch` with context `"tuple index"`). **Immutable** — no element write (`t[0] = e` is a `NotIndexable` error) — but the whole `var` can be re-assigned; value semantics (Clone deep-copies). Element coercion is element-wise, so `(1, 2)` fits `Tuple[Float64, Float64]`. `Ty::Tuple(Vec<Ty>)` / `Value::Tuple(Vec<Value>)` / `Expr::TupleLit`; the `Tuple[...]` annotation resolves via `tuple_type` in `ty_from_anno`. **Tuple unpacking** — the bare form `a, b = t` (`Stmt::Unpack { targets, value }`, a top-level comma in the target list marking the statement; each target a `NAME` or place, trailing comma allowed) — is **parsed and grammar-documented but the checker flags it `Unsupported`** (element-splitting semantics deferred; parse-only per [[syntax-first-vm-compiler-roadmap]]). Only the bare form: `var a, b = t` is a still-open Mojo feature request ([modular#2105](https://github.com/modular/modular/issues/2105)), and `for a, b in …` isn't in the docs — neither is valid Mojo, so per the strict-subset rule neither is parsed. Deferred: element write, unpacking *semantics*, and `for` over a tuple.
- **Member write / places**: an assignment target is a `NAME` (variable re-assign) or a **place** — a field/index chain rooted at a variable: `p.x = e`, `xs[i] = e`, and any nesting (`p.items[i].x = e`, `o.inner.v = e`). A place write mutates the **root variable's binding in place** (value semantics: only that binding changes; a copy stays independent). The checker (`check_place`) requires the root to be a mutable location — any variable, or `self` inside a `mut self` method (`ImmutableSelf` otherwise); writing through a call result (`mk().x = e`) is an `InvalidAssignTarget`. This same "writable place" rule governs *mutating method* receivers: a `List` mutator or a `mut self` method may be called on any place (so `bag.items.append(e)` and `p.child.increment()` work), never a temporary. A place's final step may also be a **SIMD lane** (`v[i] = e`, `vec.data[i] = e`) — a lane isn't a `Value` slot, so `update_place` special-cases it. Augmented assignment (`p.x += e`, `v[i] += e`) shares this machinery. Deferred: chained assignment (`a = b = c`).
- **Exceptions**: `raise <expr>` raises an error — an `Error` (from the built-in `Error(msg)` constructor) or a `String` (auto-wrapped). `try:` / `except [e]:` / `else:` / `finally:` handles them: one `except` (optionally binding the error name; no `except T as e:`), `else` runs only if the body didn't raise, `finally` always runs; at least one of `except`/`finally` is required. Functions carry an optional **`raises`** effect (`def f() raises -> Int:`, `raises T` typed form parsed and the type discarded — a single `Error` type is modeled). **The raises-*effect analysis is deferred** (the tough semantic choice, per the design note): the checker parses and records `raises` but does *not* require a call to a raising function to be inside a `try` or in a `raises` function. At runtime a `raise` becomes `RuntimeError::Raised(msg)` that propagates via `?` (so it crosses function-call boundaries automatically) until a `try` intercepts it or it reaches the top (an "unhandled error"). Re-raise a caught error with the **transfer sigil** `raise e^`. The `^` transfer sigil is **parsed anywhere a primary can appear** and is now **modeled as a real move** by the ownership analysis (`check_ownership`): `x^` (and the partial `p.a^`) transfers the value, so a later use of a moved place is a compile-time **use-after-move**; the VM `mem::take`s the slot at runtime, leaving a tombstone. See the ownership-analysis part of the Architecture section.
- **`with` statements (context managers, parsed — semantics deferred)**: `with item (',' item)*: <block>`, each `item` a context `expression` with an optional `as NAME` (`with open(p) as f:`, `with lock():`, `with a() as x, b() as y:`). `Stmt::With { items: Vec<WithItem { context, var }>, body }`; the `with` keyword is a new `Token::With` (Python-shared), parsed by `parse_with` (comma-separated items, optional `as NAME`). Per [[strict-subset-of-mojo]] only the **bare** forms verified in the Mojo docs are parsed — the `as` target is a plain `NAME`, and the **parenthesized** (`with (a as x, b as y):`) and **tuple-target** forms (absent from the docs) are not. The checker flags it `TypeError::Unsupported("with statement")` (the `__enter__`/`__exit__` protocol is deferred — "parse now, run later" per [[syntax-first-vm-compiler-roadmap]]); evaluator arm is a safety net.
- **SIMD**: `SIMD[DType.<dt>, N]` is a built-in fixed-width vector type (built on Phase 2 value parameters — `N` is a comptime power-of-two `Int`). Element types `<dt>`: `int8/16/32/64`, `uint8/16/32/64`, `float32`, `float64`, `bool` (wider ints/low-precision floats deferred). **Scalar aliases** `Int8`…`Int64`, `UInt8`…`UInt64`, `Float32` mean `SIMD[dtype, 1]`; **`Float64` is unified with `SIMD[DType.float64, 1]`** — the same type (as in real Mojo), so a `Float64` splats into a `float64` vector and a `float64` lane reads/writes as a `Float64`. The unification is *type-level*: `SIMD[DType.float64, 1]` canonicalizes to `Ty::Float64` (checker `simd_ty`) and a width-1 `float64` value to `Value::Float64` (evaluator `simd_value`), so width-1 `float64` keeps its native scalar representation and arithmetic path (`Int`/`UInt`/`Bool` remain separate non-SIMD types). Construction is variadic (`SIMD[DType.int32, 4](1, 2, 3, 4)`) or **splat** (`SIMD[DType.int32, 4](7)`, `Int32(5)`); operators are **elementwise** (`+ - *` on numeric lanes, `/` on floats, comparisons `== != < > <= >=` → a `SIMD[DType.bool, N]` mask), with a numeric *literal* splatting to the other operand. A lane **read** `v[i]` (`i: Int`) returns the width-1 scalar; a lane **write** `v[i] = e` (and `v[i] += e`) sets lane `i` from a same-dtype scalar or a splatting literal — reachable through a field too (`vec.data[i] = e`), and wrapped to the element width like construction (see the member-write note). Integer arithmetic is **bit-accurate** — it wraps at the element width (`wrap` in `evaluator.rs`: int8 `200 → -56`, uint8 `255 + 1 → 0`); `float32` rounds each result to single precision (`round_f32`). `DType.<dt>` is a built-in namespace usable only inside a `SIMD[...]` argument. Deferred: `// % **` on SIMD, mixed-width broadcast, and a value-parameter (symbolic) width.
- **Traits (Phase 1b — bounded generics)**: a `trait Name:` declares **method requirements** — `def m(self, …) -> ret:` headers whose body is exactly `...` (the ellipsis). A struct declares **conformance** with a parenthesized trait list after its name (and after any `[type_params]`): `struct Duck(Copyable, Quackable):`. Conformance is **explicit/nominal** (Mojo-style): the checker verifies the struct implements every method a *user* trait requires with a matching signature (`MissingTraitMethod`/`TraitMethodMismatch`); **built-in** traits impose no checked requirement (any type "conforms", so we needn't model, e.g., what `Copyable` requires). `Self` is a type: bare **`Self`** is the enclosing struct type (in a struct method) or the abstract conforming type (in a trait requirement, checked as `Ty::SelfType` and substituted at conformance/use). A trait-bounded parameter `[T: Quackable]` makes `x.quack()` type-check via the trait's requirement (with `Self` = `T`); at a call site, after `T` is solved to a type `S`, the checker verifies `S` **conforms** to every bound (`TraitNotSatisfied`) — a bounded `T` itself conforms to a trait in its own bounds, so it can be forwarded to another `[U: Quackable]` parameter. Everything is still **type-erased at runtime**: a `trait` produces no runtime binding, `x.quack()` is an ordinary method call on the conforming struct value (which already carries the method), so no trait dispatch machinery exists in the evaluator. Three further valid-Mojo trait forms **parse** (full AST + grammar) but the checker flags each `TypeError::Unsupported`: **trait inheritance/refinement** (`trait Bird(Animal, Named):` — `Stmt::Trait.refines`, reusing `parse_conformance`), **default method implementations** (a real body instead of `...` — `TraitMethod.default_body`, parsed via `parse_block_body` after peeking past the `INDENT` for `...`), and **`comptime` members** (`comptime count: Int` — `Stmt::Trait.comptime_members`, `TraitComptime { name, ty }`). `check_trait` flags a trait using any of these; a *pure* trait (only `...` methods, no `refines`, no comptime members) is unchanged. **Generic traits** `trait T[U]:` are **not** valid current Mojo (only conforming structs are parameterized), so per [[strict-subset-of-mojo]] no trait `[type_params]` is parsed. Still deferred (semantics): the three flagged forms above, plus trait runtime fields (Mojo has none).
- **Closures**: a nested `def` captures its enclosing locals (read *and* write). **Downward funargs only** — a closure may be called by name in its defining scope, but may NOT escape (be returned/stored beyond its scope); the checker enforces this (`TypeError::ClosureEscape`), matching Mojo (no escaping closures). The VM implements a nested `def` by **lifting it to a mangled top-level function** whose captured locals become leading `mut` parameters (so reads and writes both work — reference-capture via the existing write-back), and rewriting calls to prepend those captures; self-recursion works. Refused cleanly (VM `Unsupported`): a nested `def` that calls a *sibling* nested `def`, deeper nesting, and generic nested defs. Do not "fix" this into full Scheme-style first-class closures without an explicit decision — the downward-only restriction was chosen deliberately.
- **Literals/expressions**: integers — decimal plus **bases** `0xFF`/`0o77`/`0b0111` and **`_` digit separators** (`1_000_000`, `0xFF_00`; stripped before parsing) — floats, `True`/`False`, strings in **single or double quotes** and **triple-quoted** multi-line (`"""…"""`/`'''…'''`; the **full Mojo escape set** — simple `\a \b \f \n \r \t \v \\ \' \"` plus numeric `\ooo` octal / `\xHH` / `\uHHHH` / `\UHHHHHHHH`, each a Unicode scalar UTF-8-encoded; no `\u{…}` brace form — see `decode_escape`), `None`, identifiers, parenthesized grouping, and call expressions. **t-strings** `t"…{expr}…"` / raw `rt"…"` (Mojo's interpolation — *not* f-strings; verified against docs) lex into text + interpolation chunks (`Token::TString`), and the parser parses each `{…}` into a real sub-expression (`Expr::TString { parts: Vec<TStringPart::{Literal,Expr}>, raw }`); the checker flags a t-string `Unsupported` (semantics deferred). Numeric bases/separators and single/triple strings are **fully supported** (ordinary `Int`/`String` values that run).
- **Walrus / named expression** `name := value` (`Expr::Named`): the assignment-expression from Mojo/Python (binds looser than every operator; target is a bare name). **Parsed** and type-checked (as `value`'s type) but the evaluator flags it as `RuntimeError::Unsupported` — the "parse now, run later" strategy for a valid-Mojo construct we don't run yet (like `var`-less introduction). The reusable mechanism is `RuntimeError::Unsupported(msg)`; the checker passes these through so the *evaluator* is the single place that reports "unsupported feature".
- **Conditional expr / chained comparison / slices (parsed, semantics deferred — syntax-first phase, see [[syntax-first-vm-compiler-roadmap]]):** the **ternary** `a if c else b` (`Expr::IfExpr`; precedence `Conditional`, looser than `or`, tighter than the walrus; the else-branch nests right); **chained comparison** `a < b < c` / `0 <= i < n` (`Expr::Compare { first, rest: Vec<(InfixOp, Expr)> }` — a chain of ≥ 2 comparisons, each operand parsed once; a *single* comparison stays an `Expr::Infix`, so nothing regresses — built in `parse_infix` via `peek_is_comparison`/`parse_comparison_op`, which also fold in `in`/`not in`); and **slice subscripts** `a[i:j:k]` with every bound optional (`a[:j]`, `a[::2]`, `a[i:]` — `Expr::Slice { object, lower, upper, step }`, parsed in the overloaded postfix `[` via `parse_slice_rest`, distinguished from a generic call / plain index by a top-level `:`). All three **parse into the AST + grammar** but the *checker* flags them `TypeError::Unsupported` (evaluator arms are a safety net) — pure syntax coverage, no semantics.
- **Operators** (Pratt-parsed): unary `-` and `not`; binary `+ - * / // % **`, comparisons `== != < > <= >=`, membership `in` / `not in` (comparison-level → `Bool`; `x in List[T]`/`String` — see the List bullet), and short-circuiting `and`/`or`. Precedence (low→high) is `walrus < conditional(ternary) < or < and < not < comparison < +,- < *,/,//,% < unary - < ** < call`, matching Python/Mojo; `**` is right-associative and binds tighter than unary `-` (`-2 ** 2 == -(2**2)`). Numeric ops work across `Int`/`UInt`/`Float64` (with literal coercion / promotion of mixed literal+concrete operands): `+ - * // % **` return the operand type, `+` also concatenates `String`, ordering `< > <= >=` → `Bool`; `/` is true division returning `Float64` for any numeric operands (`Int / Int → Float64`); `//`/`%` use Python flooring (result takes the divisor's sign; integer `//`/`%` by zero is a runtime error); `**` integer exponents must be non-negative; unary `-` is `Int`/`Float64` (not `UInt`). Equality covers the numeric types plus `Bool`/`String`/`None`.
- **Control flow**: `if`/`elif`/`else`, `while`, `for var in <iterable>`, and `break`/`continue`. Each header ends in `:` and an indented block (no inline form); conditions must be `Bool`. Each control-flow block is its own lexical scope (so the `for` loop variable does not leak, and a `var` in a branch is scoped to it). `break`/`continue` outside a loop are rejected by the checker (`TypeError::BreakOutsideLoop`/`ContinueOutsideLoop`), as `return` outside a function is.
- **`print(...)`** is the output facility: a built-in taking any number of *printable* arguments (everything except functions, ranges, and opaque type parameters — `is_printable` in `checker.rs`), which it writes via their `Display`, space-separated with a trailing newline, returning `None`. No keyword arguments (`sep=`/`end=` unsupported), and unlike Mojo it doesn't require `Writable` conformance. The VM **buffers** output (readable via `Backend::output()`) rather than writing to stdout, so it's testable; the CLI `run` prints the captured buffer. Other built-ins: **`String(x)`** stringifies a numeric/`Bool`/`String` (via `Display` — so `Bool` is `true`/`false`, a display convention) → `String`, enabling `"n = " + String(n)`; **`abs(x)`** (numeric → same type), **`min(a,b)`/`max(a,b)`** (two numerics unified like an operator → common type), **`round(x)`** (`Float64` → nearest `Float64`, ties half away from zero), **`len(s)`** (`String` byte length → `Int`).
- **`range(...)`** is the only iterable: a built-in (`range(stop)`, `range(start, stop)`, `range(start, stop, step)`) yielding `Int`s over the half-open `[start, stop)`. It is *not* first-class — there is no `range` type annotation, so a range only lives in a `for` header. A zero `step` is a runtime error (`range(...)` step can be a runtime value, so the checker can't catch it). `range` is resolved as a built-in in call position only when not shadowed by a user binding; it is handled in both the VM and the checker (`infer_call`).

- **Static type checking** (`src/checker.rs`): a pass between the parser and the compiler backend enforces the annotations the backend relies on. It is *sound but not complete* — if `check` succeeds, execution won't raise `UndefinedVariable`/`TypeError`/`NotCallable`/`ArityMismatch`/`ClosureEscape` (the exceptions: a `range(...)` zero `step`, a SIMD lane index out of range, a **deliberate `raise`** — `RuntimeError::Raised` — and the handful of parse-only constructs the VM still flags `RuntimeError::Unsupported`, e.g. the walrus `:=`, which the checker deliberately passes through) — it *also* rejects, at check time, parsed-but-unimplemented **function-argument forms** (default values, `*args`/`**kwargs`, `/`/`*` markers, argument conventions, keyword call-args) via `TypeError::Unsupported`, since their bodies/calls can't be meaningfully typed without the semantics. It catches at compile time several things Mojo also rejects statically: var-init/argument/return type mismatches, bad operands, non-`Bool` `if`/`while` conditions, a `for` over a non-`range`, escaping closures, same-scope redeclaration (the evaluator silently overwrote), `return`/`break`/`continue` outside their enclosing construct, an assignment with a value of the wrong type, and a non-`None` function that can **fall off the end** without returning (path-sensitive — `MissingReturn`; see `definitely_returns`). See the checker section below.

### Planned direction (not yet implemented)

- **Generics Phase 2 — mostly done** (value parameters, comptime `Int` arithmetic, `comptime` constants). `comptime if`/`comptime for` now **parse** (checker flags them `Unsupported` — semantics deferred). Remaining comptime work: value parameters of non-`Int` type, `comptime if`/`comptime for` *semantics* (branch selection / unrolling), and dependent types (a value parameter appearing in a field/param/return type, e.g. `-> FixedBuffer[n]`). Smaller trait follow-ups also deferred: default method implementations (a real body instead of `...`), trait fields, generic traits, and trait inheritance.
- **SIMD — core done** (see the SIMD bullet above); `Float64`≡`SIMD[DType.float64,1]` unification landed. Deferred: wider-integer/low-precision-float dtypes, `// % **` on SIMD, mixed-width broadcast, and value-parameter (symbolic) widths. Unifying the existing scalars (`Float64` etc.) with `SIMD[_, 1]` — as real Mojo does — is a larger refactor left undone.
- **Function/closure types** `def(T1, …) [thin] [raises] -> R` **are parsed** (`Type::Func`; parameters are types only, `thin` = non-capturing, `abi("…")` parsed and discarded — verified against current Mojo docs) but their *semantics are deferred*: `ty_from_anno` flags any function-typed binding as `TypeError::Unsupported` (the checker's internal `Ty::Func` still only ever arises from a real `def`, carrying names/arity). Capture lists `{…}` on a closure are **not** parsed (their current-Mojo grammar is unsettled — the `unified` keyword was removed; do not invent it).
- **Forward references / mutual recursion across sibling `def`s.** The checker binds names sequentially (matching the evaluator) and checks each `def` body at its definition site, so a body can't call a sibling `def` declared later in the same block. Fixing this needs per-block hoisting of `def` signatures.
- **Return checking is now path-sensitive** (`definitely_returns` in `checker.rs`): a non-`None` function must return on every path (`MissingReturn`). It is intentionally conservative — an `if` counts only with an `else` where every arm returns, and a loop never counts (may run zero times; `while True` is not special-cased), so a few valid programs are rejected.

## Architecture

The surface syntax is documented in `grammar.md` — a PEG grammar (in the style of CPython 3.16's reference grammar, restricted to Mojo) covering everything implemented. Update it *before* adding syntax.

The pipeline is **source → `Lexer` (tokens) → `Parser` (AST) → `Checker` (typed AST) → `hir::Cfg` (control-flow graph) → `mir` (A-Normal-Form MIR) → `analysis` (ownership/move + liveness/drop) → `backend::VmBackend` (register VM, over the verified MIR)**. `Token` lives in `src/token.rs`, the AST in `src/ast.rs`, and the runtime `Value` + value operations in `src/runtime/mod.rs` — every stage shares them. The **frontend** (lexer/parser/checker/HIR/MIR) is backend-agnostic; the **backend** sits behind a `trait Backend` (`src/backend/mod.rs`) with a `BackendKind` selector, so a future native backend (e.g. Cranelift) can consume the same verified MIR. `src/main.rs` drives the CLI.

### Lexer (`src/lexer.rs`) — the core

The lexer is the heart of the project. It implements Python's **offside rule** (significant indentation) as an `Iterator<Item = Result<Token, LexError>>`, so consumers pull tokens lazily. The complexity lives entirely in `Lexer::next`, a single state-machine loop. Key state on the struct:

- `indent_stack: Vec<usize>` — stack of active indentation column widths (starts `[0]`). Increasing indent pushes and emits `Indent`; decreasing pops and emits one `Dedent` per level unwound. An indent that doesn't match any stack level is an `IndentationError`.
- `paren_count` — while `> 0`, indentation and newlines are **suppressed** (implicit line continuation inside `( ... )`). **Explicit line continuation** is also supported: a `\` immediately before a newline (`LF`/`CRLF`) is consumed with no `Newline` emitted and `at_line_start` left false, so the continued line joins the current logical line (its indentation not significant) — matching Mojo's `line_continuation` token; a `\` not followed by a newline is an `UnexpectedCharacter` error.
- `at_line_start` — gates the leading-whitespace/indentation logic so it runs once per logical line; blank lines are skipped without emitting tokens.
- `pending_tokens: VecDeque<Token>` — because one input position can produce several tokens (e.g. multiple `Dedent`s, or the `Newline` + `Dedent`* + `Eof` burst at end of input). The loop always drains this queue before scanning more input.

**Comments** (`#` to end of line) are lexed away, never tokenized (`skip_to_line_end` advances by whole chars, so multibyte comment text is safe). A comment-*only* line (spaces then `#`) is detected in the at-line-start block and treated exactly like a blank line — no `Newline`/`Indent`/`Dedent`, so comments at any indent are inert; an inline `#` is a normal step-5 arm that skips to EOL, leaving the trailing `Newline` (and a `#` inside `( … )` is skipped like other continuation whitespace).

At EOF the lexer synthesizes a trailing `Newline` (if needed), unwinds the indent stack to emit closing `Dedent`s, and emits exactly one `Eof` (guarded by `eof_emitted`).

Number scanning (`consume_digit_run(radix)` — digits with `_` separators between digits — plus `consume_exponent`) reads a based prefix (`0x`/`0o`/`0b` → `i64::from_str_radix`) or decimal digits, then decides `IntLiteral` vs `FloatLiteral`: a `.` followed by a digit, or an `e`/`E` exponent, makes it a float (a bare trailing `.` is left alone, reserved for a future member-access `.`); `_` separators are stripped before parsing. String scanning (`lex_string(quote, triple)`) handles single/double and triple-quoted (multi-line) strings; the shared `decode_escape` helper (used by `lex_string` and the non-raw `lex_tstring` path) decodes the full escape set — simple `\a \b \f \n \r \t \v \\ \' \"` and numeric `\ooo`/`\xHH`/`\uHHHH`/`\UHHHHHHHH` (via `read_hex_digits`/`read_octal_digits`), each a Unicode scalar (a surrogate/out-of-range value or bad digit is a `LexError::InvalidEscape`). **t-strings** (`lex_tstring`): a `t`/`rt` word immediately before a quote lexes a `Token::TString { chunks: Vec<TStringChunk::{Text,Interp}>, raw }`, scanning `{…}` interpolations (brace-depth aware, `{{`/`}}` literal); the parser's `build_tstring`/`parse_interpolation` re-lex+parse each `Interp` fragment into a real `Expr`.

### Token (`src/token.rs`) and AST (`src/ast.rs`)

`token.rs` is the single source of truth for the `Token` enum (used by `lexer.rs`, `parser.rs`); generics added `LBracket`/`RBracket` (`[` `]`, which nest like parens for the offside rule) and `Amp` (`&`, trait-bound conjunction), traits added the `Trait` keyword and `Ellipsis` (`...`), value parameters added the `Comptime` keyword, exceptions added `Raise`/`Try`/`Except`/`Finally`/`Raises` keywords and `Caret` (`^`), and imports added `Import`/`From`/`As` keywords. The keyword variants are grouped **Mojo-only** (`var struct trait comptime raises`) vs **shared-with-Python** (the rest), mirroring the reference tree-sitter Mojo grammar's split; `Token::keyword(&str) -> Option<Token>` is the one reserved-word table (the lexer calls it after scanning a word, falling back to `Identifier`). Augmented-assignment tokens (`PlusEq`…`DoubleStarEq`) and `ColonEq` (`:=`, walrus) round out the operator set. Soft words like `mut` are intentionally *not* in it (so `mut self` parses contextually and `mut` stays a usable name). `ast.rs` holds the tree the parser produces and the evaluator/tests consume: `Type` (incl. `Named(String, Vec<ParamArg>)` for parameterized types, `SelfParam(String)` for `Self.T`, `SelfType` for bare `Self`, and `Func { params, ret, thin, raises }` for a function type `def(…) -> …`), `ParamArg` (`Type(Type)` or `Value(Expr)` — a supplied parameter argument), `TypeParam { name, bounds }` (a declared `[name: X]`, classified later), `Dtype` (a SIMD element type), `Param` (a struct field: name + type), `FnParam` (a function/method parameter: name, type, `default`, `kind` (`ParamKind::{Regular,Variadic,KwVariadic}`), `convention` (`ArgConvention`)), `KwArg` (a `name=value` call argument), `Decorator` (`{ path: Vec<String>, args, kwargs }` — a general `@dotted.name[(args)]`), `Method` (a struct method; `self` is implicit, not in `params`; carries `has_self`, `self_convention` (`Option<ArgConvention>`), `decorators`, `raises`, and `positional_only`/`keyword_only` marker indices), `TraitMethod` (a `...` requirement), `Stmt` (incl. `Struct` with `type_params`/`conforms`, `Def` with `type_params`/`raises`, `Trait`, `Comptime`, `Raise`, `Try`, `Import`, `FromImport` with `ImportNames`/`ImportName`, `SetPlace { place, value }` for a member/index write `p.x=e`/`xs[i]=e`, and `AugAssign { place, op, value }` for `target OP= e`), `Expr` (incl. `Call` with `param_args`, `Member`, `MethodCall`, `Index` for `v[i]`, `Transfer` for `x^`, `ListLit` for `[a, b, …]`, and `Named { name, value }` for the walrus `name := value`), and the operator enums `PrefixOp`/`InfixOp`. No new tokens were needed for SIMD (`[` `]` `.` already exist). Keeping the AST separate from the parser lets future consumers (a type checker, tests) depend on the tree without pulling in parsing. Add new node kinds here.

### Parser (`src/parser.rs`)

Statements are recursive-descent; **expressions are a Pratt parser** (`parse_expression(min_precedence)` with precedence climbing). It is generic over any `Iterator<Item = Result<Token, LexError>>`, so it consumes the `Lexer` directly (`Parser::new(Lexer::new(src))`), using one-token lookahead via the `Peekable` adapter, and surfaces lexer errors as `ParseError::LexerError`. Key pieces: `parse_prefix` handles literals/identifiers/grouping and the unary operators; `parse_infix` handles binary operators and the postfix call `(...)`; `peek_precedence` + the private `Precedence` enum drive the climbing loop. Calls are by name (`Expr::Call { name, .. }`) — the callee must be an identifier, which is enough since closures can't escape to become arbitrary callee expressions. Left-associativity comes from parsing each binary operator's right operand at the operator's own precedence (`infix_precedence`). Generics/traits live only in *type* and *header* position, never in expression position: `parse_type_params` reads the optional `[T: Bound & Bound, …]` after a `struct`/`def` name (a bound is required — Mojo has no unconstrained parameter), `parse_conformance` reads a struct's optional `(Trait, …)` list, `parse_trait`/`parse_trait_method` read a `trait` and its `...`-bodied requirements, and `parse_type` reads `Self.T`, bare `Self`, and a named type's optional `[type args]` (`parse_type_args`). Explicit `[...]` **is** allowed at a call/construction site (`Name[param_args](args)` — Phase 2), handled as a postfix at `Call` precedence in `parse_infix`; `parse_param_arg` disambiguates each bracket argument (a leading type keyword / `Self` is a `Type`; an identifier followed by `[` is a parameterized type; otherwise a value `Expr`, reinterpreted as a type by the checker if the parameter is a type one). `parse_comptime` handles `comptime NAME = expr`. A postfix `[` is three-way: after parsing the bracket, `(` following ⇒ generic call; nothing ⇒ a **subscript** `Expr::Index` (SIMD lane read, a single value entry, any object). `parse_raise`/`parse_try` handle exceptions (`parse_try` requires at least one of `except`/`finally`); `parse_raises_effect` reads the optional `raises [T]` on a def/method (the type is discarded); `^` is a postfix `Expr::Transfer` at `Call` precedence. `parse_import`/`parse_from_import` handle imports (`parse_dotted_name` for the module path; relative-import levels accumulate from `Dot` (1) and `Ellipsis` (3) tokens; the checker/evaluator have no-op arms).

### Checker (`src/checker.rs`)

Static type checker that runs after parsing, before the compiler backend (`check(&program)` / `Checker`). It walks the AST once with a scope chain (`Vec<HashMap<String, Ty>>`, innermost last) mirroring lexical scoping, inferring an internal `Ty` for every expression and verifying it against the context. (Where the text below says the checker "mirrors the evaluator", read "mirrors the runtime semantics in `src/runtime`", which the VM shares.) `Ty` extends the scalar `ast::Type` with a `Func { params, ret }` variant synthesized from each non-generic `def`, a `GenericFunc { type_params, params, ret }` from a generic `def`, a `Param { name, bounds }` (a type parameter inside a generic body, carrying its trait bounds), `SelfType` (a trait requirement's abstract `Self`), and `Struct(name, Vec<Ty>)` carrying type arguments (the annotation grammar has no function-type syntax, so `Func`/`GenericFunc` never come from an annotation). `infer`/`infer_prefix`/`infer_infix`/`infer_call` reproduce the evaluator's operator and call typing *exactly* — same `Int`/`String`/`Bool` operand rules, same equality-between-equal-scalars rule — so the two stages can't disagree. `check_block` threads two pieces of context: the enclosing function's declared return type as `ret: Option<&Ty>` (`None` = "not in a function", so a top-level `return` is `ReturnOutsideFunction`) and `in_loop: bool` (gating `break`/`continue`; a nested `def` resets it to `false`). `declare` binds in the innermost scope and rejects same-scope redeclaration; a `def` name is declared in the *enclosing* scope before its body is checked (enabling recursion), then params are declared in a fresh pushed scope. Before any of that, `advanced_param_feature` gates a `def`/method/trait signature that uses a parsed-but-unimplemented parameter form (`*args`/`**kwargs`, convention, `/`/`*` marker) → `TypeError::Unsupported`; `reject_kwargs` still flags keyword args on a *method* call. **Default parameter values are implemented** for non-generic free `def`s (so `advanced_param_feature`'s `flag_defaults` arg is `false` on the `Stmt::Def` path, `true` for methods/traits which still defer them): `required_count` computes the leading required-arg count (rejecting a required param after a defaulted one), each default is type-checked against its parameter, and the resulting `Ty::Func` carries `required` (and parameter `names`). **Keyword arguments** to a free function are matched by `infer_call` via the shared free fn `match_call_slots(names, nparams, required, npos, kw_names, has_variadic) -> (Vec<ArgSlot>, Vec<usize>)` (`ArgSlot::{Positional,Keyword,Default}`; the second vec is the overflow positional indices a **`*args`** collects), which the evaluator reuses so the stages bind identically; its `MatchError` maps to `TypeError::BadCall` (unknown keyword / duplicate / missing required) or `ArityMismatch` (too many positional). Keyword args to a built-in, struct construction, or generic function are rejected. **`*args`**: `Ty::Func` carries an optional `variadic` *element* type (from a trailing `ParamKind::Variadic`); the regular params/`names`/`required` exclude it. `infer_call` checks each overflow argument coerces to the element type; the body binds the variadic parameter as `Ty::List(element)`. A plain or defaults-only signature (`param_tys` reads each `FnParam.ty`) is otherwise unaffected. `Stmt::Assign` to an in-scope name requires the value's type to equal the target's and rejects `Func`-valued assignment as a `ClosureEscape`; to an **out-of-scope** name it is a `var`-less introduction — the checker binds the implicit var (with the value's `default_literal` type) so the rest type-checks, and the VM runs it as an ordinary binding. `Expr::Named` (walrus) infers as its value's type but does not bind the name (`infer` is read-only). `check_scoped_block` pushes a fresh scope for each `if`/`else`/loop body (so control-flow blocks scope their `var`s and the `for` loop variable). Returning a `Func` is a `ClosureEscape`. `Ty` also has a `Range` variant (from the built-in `range`, the only iterable); `for` requires its iterable to be `Range` and binds the loop variable as `Int`. Structs add `Ty::Struct(name, args)` plus a `structs` registry (`name → StructInfo` of type parameters + field types + method signatures + `fieldwise_init`), separate from the value `scopes`; `ty_from_anno` validates `Type::Named` against it (name and type-argument count). A `struct` is registered (fields first, then method signatures) *before* its method bodies are checked, so methods may reference the struct's own type and call siblings; a method body is checked with `self: Ty::Struct(name, own-params-as-Param)`. `infer_member`/`infer_method_call` type field access and method calls (substituting the receiver's type arguments for the struct's parameters), and a `Name(args)` call whose name is a struct routes to `infer_construction` (fieldwise). **Generics (Phase 1a).** Two extra context stacks drive type-parameter resolution: `tparams` (a generic `def`'s parameters, so a bare `T` annotation resolves to `Ty::Param`) and `self_params` (the enclosing struct's parameters, so `Self.T` resolves — a bare `T` inside a struct is an `UnknownType`). `validate_type_params` requires each `[T: …]` to name a `BUILTIN_TRAITS` trait (user traits are deferred); the bound is then an opaque marker, so a `Ty::Param` supports no operator/method (`infer_infix`/`common_numeric` treat it as non-numeric → `BadOperator`). There is *no* explicit `f[T](...)` call syntax: `solve_type_params` infers each parameter by `unify`ing the declared param/field patterns against the argument types, then `substitute`s the solution into the checked param/return types. `unify` **defaults a numeric-literal argument to its concrete type before binding a parameter** (`IntLiteral→Int`), which forbids cross-argument literal widening (`Pair(1.0, 2)` is a type-parameter *conflict*, not `Pair[Float64]`) — this keeps the solved `Ty` equal to what the type-erased evaluator actually produces, preserving soundness. `infer_generic_call`/`infer_construction` share this machinery; `GenericFunc`, like `Func`, is a `ClosureEscape` if assigned/returned. **Traits (Phase 1b).** A `traits` registry (`name → TraitInfo` of required `MethodSig`s, whose `Self` is `Ty::SelfType`) sits beside `structs`. `check_trait` resolves each requirement with `self_ty = Some(Ty::SelfType)`; `check_struct` records a struct's declared `conforms` list and, after its method sigs are known, `verify_conformance` checks each *user* trait's requirements against the struct's methods (substituting `Ty::SelfType` → the struct type via `substitute_self`), yielding `MissingTraitMethod`/`TraitMethodMismatch`. A type parameter's `bounds` (now on `Ty::Param`) drive two things: `infer_method_call` on a `Ty::Param` receiver resolves the method from its bound traits (`lookup_trait_method`, `Self` → the `Param`), and `solve_type_params` calls `conforms_to` on each solved argument to enforce the bound (`TraitNotSatisfied`) — where `conforms_to` treats built-in traits as always satisfied, a struct as conforming iff it declared the trait, and a `Param` as conforming iff the trait is among its bounds (so a bounded `T` forwards to another `[U: Trait]`). Bare `Self` in a struct method resolves (via `self_ty`) to the struct's own type; conformance names are validated by `check_trait_name` (built-in or user trait). **Value parameters + comptime (Phase 2).** `classify_params` splits a `[...]` list into `ParamDecl::{Type{name,bounds}, Value{name}}` — a lone bound naming a scalar type (`scalar_type_name`) is a value parameter (must be `Int`). A struct type carries `Vec<TyArg>` (`TyArg::{Ty(Ty), Val(CtVal)}`, `CtVal::{Int(i64), Param(String)}`), so `FixedBuffer[8] != FixedBuffer[9]`; value arguments are concrete `Int`s (value params never appear in manipulable types, so `substitute`/`unify`/`struct_subst` only touch the `Ty` args). `eval_ct` evaluates a comptime `Int` expression (literals + `comptime` constants from the `comptimes` registry + arithmetic); `resolve_use_params` handles a use site — explicit `param_args` are resolved positionally per declared parameter (`resolve_param_arg`: a type parameter takes a bound-checked type, a value parameter a comptime `Int`), while an omitted list infers type parameters (a value parameter can't be inferred → `CannotInferTypeParam`). Value parameters are ordinary `Int` locals in a function body; a struct's are read as `Self.n` (special-cased in `infer_member`). New errors: `BadValueParamType`, `NotComptime`. **SIMD.** `Ty::Simd { dtype, width }` (width concrete — a symbolic value-param width is deferred). `ty_from_anno` resolves `SIMD[DType.<dt>, N]` (`simd_dims` → `dtype_from_arg` reads the `DType.<dt>` member + `simd_width` a power-of-two comptime `Int`) and the scalar aliases (`Dtype::from_scalar_alias` → `Simd { .., width: 1 }`). **Float64 unification**: `simd_ty(dtype, width)` canonicalizes a width-1 `float64` to `Ty::Float64` (so `SIMD[DType.float64, 1]` *is* `Ty::Float64`); it wraps every Simd-type result (construction, lane read/write target, `infer_simd_infix`), and `splats_to` accepts `Ty::Float64` for a `float64` dtype (so a `Float64` scalar splats into a `float64` vector). `float64` width-1 never becomes a `Ty::Simd`, so width-1 `float64` arithmetic stays on the scalar `common_numeric` path. Construction routes through `infer_call`'s builtin branch (`infer_simd_construction`/`infer_simd_alias_construction` → `check_simd_args`: `width` elements or 1 to splat, each `splats_to` the dtype). `infer_simd_infix` types elementwise ops (both operands the same `Simd`, or a numeric literal splatting) → same type for `+ - *` / float `/`, a `bool` mask for comparisons. `infer_index` types `v[i]`. New errors: `BadDtype`, `BadSimdWidth`, `SimdArity`, `NotIndexable`. **Exceptions.** `Ty::Error`; `raise`'s operand must be `Error`/`String`; `Error(msg)` is a builtin (`infer_error_construction`); `except e:` binds `e: Error` in a scoped clause; `Expr::Transfer` types as its operand (identity). `raises` is ignored (`Stmt::Def { .., raises: _ }`) — the effect analysis is deferred by design. **List.** `Ty::List(Box<Ty>)`; `ty_from_anno` resolves `List[T]` (`list_type`); construction (`infer_list_construction`: explicit `List[T]` coerces args, inferred `List(...)`/`ListLit` folds a common element type via `common_elem` + `default_literal`); `infer_index` and the `for` loop yield the element type; `infer_len` accepts `List`; `infer_list_method` types the method set — mutating (`append`/`insert`/`remove`/`pop`/`clear`/`reverse`/`extend`) require the receiver to be a **writable place** (`check_place(object)` succeeds, else `MutationRequiresVariable`), queries (`count`/`index`) don't; `remove`/`count`/`index` need `is_list_equatable(elem)`. **Member write**: `Stmt::SetPlace { place, value }` is checked by `check_place`, which recurses the field/index chain, validates its root is a mutable location (any variable, or `self` when the `self_mutable` flag is set — set only while checking a `mut self` method body, else `ImmutableSelf`; a non-place root is `InvalidAssignTarget`), and returns the place's type for the value to coerce to — a final `Index` into a `Ty::Simd` yields the width-1 lane type (`SetPlace` then allows a `splats_to` value, i.e. a same-dtype scalar or splatting literal). `MethodSig` carries `mut_self`; a `mut self` call's receiver must be a writable place (`check_place`), mirroring the runtime write-back. **Augmented assignment** `Stmt::AugAssign { place, op, value }` reuses `check_place` for the target and `infer_infix(op, place, value)` for the result type, requiring the result to `coerces` back to the place type (so `i /= 2` on `Int` fails). Membership `in`/`not in` is typed by `infer_membership` (dispatched first in `infer_infix`) → `Bool`, over a `List[equatable T]` or `String`/`String`. `range` is typed by `infer_range` as a built-in in call position (1–3 `Int` args), resolved only when not shadowed — mirroring the evaluator; the numeric conversions `Int`/`UInt`/`Float64` (`infer_conversion`), `print` (`infer_print` → `None`, gated by `is_printable`), and the utility builtins `String`/`abs`/`min`/`max`/`round`/`len` (`infer_stringify`/`infer_abs`/`infer_min_max`/`infer_round`/`infer_len`, each via `builtin_args` for arity) are likewise built-ins. Numeric literals get flexible types `Ty::IntLiteral`/`Ty::FloatLiteral`; `coerces` (only literals coerce — `IntLiteral`→`Int`/`UInt`/`Float64`, `FloatLiteral`→`Float64`) gates every binding site (var/assign/return/arg/field/construction), and `common_numeric` unifies operator operands (`infer_infix`). Two *concrete* numeric types thus never unify (no promotion of real values), but literal+concrete does. `/` yields `Float64`; `// % **` return the common type; non-numeric `==`/`!=` fall back to `is_scalar` (`Bool`/`String`/`None`), so `Func`/`Range`/structs can't be compared. **Design stance: sound, not complete** — names bind sequentially in source order (matching the evaluator), and `def` bodies are checked at their definition site, so sibling forward references / mutual recursion are rejected even though they'd run; this is deliberate (soundness over completeness) and noted in the planned-direction list. Errors are `TypeError` (in `error.rs`); they store rendered type *strings*, not `Ty`, so `error.rs` stays free of a dependency on the checker's internal lattice.

### Runtime values (`src/runtime/mod.rs`)

The shared value layer the backend consumes (originally extracted from the tree-walker so the VM does not depend on it). Owns the `Value` enum — `Int(i64)`/`UInt(u64)`/`Float64(f64)`/`Bool`/`Str`/`None`/`Struct { name, fields, value_params }`/`List(Vec<Value>)`/`Tuple(Vec<Value>)`/`Simd { dtype, lanes }`/`Error(String)`/`Range { start, stop, step }`/`Moved` (a move tombstone) — and all the **value operations** both the checker's semantics and the VM agree on: `apply_infix`/`apply_prefix` (with `numeric_op` promoting to a common `Num` kind; `floor_div`/`floor_mod` Python flooring; zero/negative-exponent guards), `coerce`/`coerce_like` (literal materialization at binding sites — there is no separate lowering pass), the `List` methods (`apply_list_method`/`list_query`), the SIMD ops (`simd_binop` bit-accurate `wrap`/`round_f32`; `read_simd_lane`/`set_simd_lane`; `simd_from_values`; the width-1-`float64`≡`Value::Float64` canonicalization in `simd_value`), the utility built-ins (`builtin_abs`/`builtin_min_max`/`builtin_round`/`builtin_convert`/`builtin_error`), `bounds_check`/`value_as_index`, `eval_membership`, `values_equal`, and `type_name`/`Display`. `Float64`'s `Display` uses `{:?}` so it keeps a decimal point.

### Compiler backend (frontend IRs + register VM)

The checked AST is lowered through two IRs, analyzed for ownership, and executed by a register VM.

**HIR / CFG (`src/hir/mod.rs`).** `Cfg::build(body)` / `build_fn(params, body)` lower a statement list into a **control-flow graph** of `BasicBlock`s on a `petgraph` `StableGraph` — control flow (`if`/`while`/`for`/`break`/`continue`/`return`/`try`) becomes blocks + edges; expressions stay nested (flattened in MIR). A `seal()` helper owns the "one terminator + out-edges per block" invariant so break/continue/return compose. `for` lowers to a while-shaped CFG driving an explicit iterator protocol (`HirInstr::{HasNext, Next}`). A `Lower.vars` interner assigns each name a `VarId` (seeded from `params` so `vars[0..n_params]` are the parameters — the call ABI). `Terminator` includes `EscapeJump(BlockId)` — a `break`/`continue` inside a `try` region that targets an **enclosing function loop** (see MIR/VM); a `LoopFrame { header, exit, escape }` stack distinguishes an in-CFG loop (a plain `Jump`) from an escapable one.

**MIR / ANF (`src/mir/mod.rs`).** `lower_program(program)` produces one `MirFunction` per top-level `def` / struct method plus a synthetic `__toplevel__`; `lower_cfg` flattens each HIR body to **A-Normal Form** — every subexpression becomes a three-address `MirInstr` writing a fresh `Reg`, with a `SpanTable` mapping each temp to its source span. Writes go through a **place** (`MirPlace` = root `VarId` + `Proj::{Field, Index}` chain; `Store`/`LoadPlace`), mirroring rustc's place/rvalue split. Key instrs: `UseVar { mode: Copy|Move|Borrow* }`, `MovePlace` (a partial move `p.a^`), `DefVar`, `Call`/`MethodCall` (carrying `kwargs`, `arg_places` for `mut`/`ref` write-back, and `param_arg_regs` for value-parameter reification), `MakeList`/`MakeTuple`/`MakeSimd`, `Raise`, `Try` (structured sub-regions), and `DropVar` (spliced by analysis). Terminators: `Jump`/`Branch`/`Return`/`FallOff` (normal end of a `try` region) / `EscapeJump { target, cleanup }`. **Nested `def`s** are lifted here: their captured enclosing locals become leading `mut` parameters and calls are rewritten to prepend them (`lower_fn_nested`, free-variable analysis).

**Ownership analysis (`src/analysis/mod.rs`).** `check_ownership(program)` runs over the MIR — a deliberately **separate compile stage** (also in the CLI `check`/`run`). *Move/init* is a forward dataflow with a **place-tree lattice** (`Node { base: Own, children }`, `Own::{Owned, Moved, MaybeMoved}`) so a partial move `p.a^` moves just that field: using `p.a` after is a **use-after-move**, using `p.b` stays fine, a whole-`p` use is flagged, a field store reinitializes. Cross-branch merges yield `MaybeMoved` → **conditional-move** diagnostics (with spans). `elaborate_drops_program` is a backward-liveness pass that splices `DropVar` at each variable's last use (**ASAP destruction**: a struct's `__del__` runs there, fields in reverse order), skipping `^`-moved and borrowed values; it inserts **edge drops** on branches, fills each `Try.cleanup` (body-locals) and each `EscapeJump.cleanup` (outer locals dying on a `break`/`continue` escape edge), and **skips `__toplevel__`** (module vars live to program end).

**Register VM (`src/backend/vm.rs`).** `VmBackend` implements `trait Backend`; `run` lowers the program (`lower_program` → `elaborate_drops_program`), builds struct/signature registries, then executes `__toplevel__` and `main()`. Each call is a frame of `regs` + `vars` (indexed by `VarId`); `call_frame` walks a `MirFunction`'s blocks. A `Flow` enum (`Normal`/`Return(Value)`/`Jump(block)`) threads non-local control out of `try` regions: `exec_try` runs the structured region (`run_region`), body-local cleanup, `except`/`else`/`finally` (a `finally` outcome overrides), and propagates a `Return` or an escaping `break`/`continue` (`Flow::Jump`) that `call_frame` then jumps on. `UseVar { Move }`/`MovePlace` `mem::take` the source (tombstone); `Store`/`LoadPlace` navigate places (`nav_step`/`store_place`, with a SIMD-lane special case); `mut`/`ref` parameters and `mut self` write back through the caller's `arg_places`/`recv_place`; value parameters are reified onto structs / bound as frame locals; `DropVar` runs `drop_value` (user `__del__` then fields). Generics and traits are **type-erased**; a `trait` produces no runtime binding (dispatch is an ordinary method call on the conforming struct). Built-ins (`print`/`range`/`String`/`len`/`abs`/`min`/`max`/`round`/`Int`/`UInt`/`Float64`/`Error`/`List`/`SIMD`) run as intrinsics via the shared `runtime` ops. Remaining clean-`Unsupported` gaps (never wrong answers): a `break`/`continue` targeting a loop declared *inside* an enclosing `try`, sibling-calling / deeper-nested / generic nested `def`s, nested `struct`/`trait`, and the parse-only constructs the checker already flags. `VmBackend::bindings()` exposes the `__toplevel__` frame's user vars for the CLI dump / tests.

### Errors (`src/error.rs`)

`LexError` (byte offsets into the source) is produced by the lexer, `ParseError` (wraps `LexError`) by the parser, `TypeError` by the checker, `OwnershipError` (use-after-move / conditional-move, with spans) by the ownership analysis, and `RuntimeError` by the VM. Each stage owns its own error type; they compose by wrapping.
